# Design docs

One file per meaningful decision or system: the *why*, not just the what
(the *what* belongs in `CLAUDE.md` or the code itself).

praxis's `CONTRIBUTING.md` has a good four-question filter for whether
something belongs here:

1. What observation or need does this enable?
2. What does it introduce or change?
3. How is the result validated?
4. How would another engineer test it?

If a doc here can't answer at least the first two, it's probably a
`docs/snapshots/` entry instead — a record of a session, not a standing
design rationale.
