Ontology-Driven Schema Evolution: LLM Policies That Survive Column Changes
dltHub Blog via TLDR Data  ·  May 13, 2026
Key Summary Points
01 The approach separates policy definition from implementation: describe rules in plain English, encode as an LLM-readable ontology, apply at runtime per column to generate analytics-safe views.
02 Crisp cases (name patterns, data types, cardinality thresholds) are handled deterministically; ambiguous cases requiring value inspection are routed to the LLM.
03 When 5 new columns were added, the identical ontology policy correctly classified all of them without code changes — including a UUID column requiring value sampling to rule out PII.
💼 Professional Reading
Key Takeaways
Ontology-based policies remain valid across schema changes where static allowlists grow stale — the policy is written once in English and adapts automatically.
Key limitation: approach cannot detect cross-column re-identification risks or numeric columns that encode sensitive inferences — human review still needed for complex PII.
Works best when policies decompose into patterns and thresholds rather than requiring complex contextual reasoning — a clear fit/no-fit boundary for this pattern.
